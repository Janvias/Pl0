# PL/0语言编译原理可视化系统
# 程序说明书

---

## 摘要

本程序说明书详细描述了PL/0语言编译原理可视化系统的设计与实现过程。该系统基于C++ 11标准，采用EasyX 2024图形库作为可视化开发工具，在Visual Studio 2022环境下开发完成。系统实现了完整的词法分析器，能够识别PL/0语言的关键字、标识符、常数、运算符和界限符；实现了LL(1)自顶向下语法分析器，支持FIRST集、FOLLOW集、SELECT集的自动计算以及LL(1)分析表的自动生成；实现了SLR(1)自底向上语法分析器，支持项目集规范族、活前缀DFA和SLR分析表的自动生成；实现了错误检测与提示功能，支持常见语法错误的检测与定位。此外，系统还实现了语法分析树可视化功能作为加分项。所有分析过程均配有图形化界面展示，使用户能够直观地观察编译原理各阶段的工作机制。

**关键词**：PL/0语言；词法分析；LL(1)语法分析；SLR(1)语法分析；可视化；EasyX图形库

## 第1章 概述

### 1.1 项目背景与意义

编译原理是计算机科学与技术专业的核心课程之一，其内容涉及形式语言理论、自动机理论、算法设计等多个领域。语法分析作为编译原理课程的核心章节，历来是学习的重点和难点。传统的教学方式侧重于理论知识的讲授，学生往往难以将抽象的文法规则、自动机理论与实际的程序执行过程联系起来，理解起来较为困难。

PL/0语言是一种经典的教学型程序设计语言，其语法结构简洁明了，便于教学和学习。通过设计并实现一个PL/0语言的编译原理可视化系统，可以将抽象的理论知识转化为直观的图形展示，帮助学生更好地理解词法分析、语法分析的工作原理。同时，通过亲手实现词法分析器、语法分析器，学生可以加深对理论知识的理解，提高动手实践能力。

本项目的意义在于：一是提供一个交互式的学习工具，使抽象的编译理论变得形象具体；二是通过可视化展示，帮助学生观察编译过程中各数据结构的动态变化；三是为后续课程设计和毕业设计奠定基础，培养学生的软件工程能力。

### 1.2 课程设计任务要求

根据课程设计任务书的要求，本项目需要完成以下必做任务和加分项：

**必做任务包括**：第一，词法分析器设计，要求能够识别PL/0语言中的关键字、标识符、无符号整数、运算符和界限符，能够过滤掉源程序中的注释和空白字符，能够进行词法错误检测并报告错误位置；第二，LL(1)语法分析器设计，要求能够自动计算文法的FIRST集、FOLLOW集和SELECT集，能够根据SELECT集自动生成LL(1)分析表，能够实现表驱动的自顶向下分析过程；第三，SLR(1)语法分析器设计，要求能够自动生成项目的闭包和GO函数，能够构造活前缀DFA，能够生成SLR(1)分析表并实现移进-归约分析。

**加分项包括**：语法分析树可视化，要求在语法分析过程中动态生成语法分析树并以图形方式展示。

### 1.3 系统总体功能介绍

本系统主要包含以下核心功能模块：

**词法分析模块**负责对源程序进行扫描，将输入的字符序列转换为Token序列。该模块实现了状态转换图驱动的词法分析算法，能够准确识别PL/0语言的各种单词符号，并记录每个Token的行号信息以便后续错误定位。

**LL(1)语法分析模块**实现了自顶向下的预测分析方法。该模块能够对输入的Token序列进行语法分析，判断其是否符合PL/0语言的文法规则。模块内部自动完成FIRST集、FOLLOW集、SELECT集的计算以及LL(1)分析表的构建，并通过栈操作模拟推导过程。

**SLR(1)语法分析模块**实现了自底向上的规范LR分析。该模块通过构造项目集规范族和活前缀DFA，构建SLR(1)分析表，并采用移进-归约策略完成语法分析。

**可视化展示模块**利用EasyX图形库实现了各分析过程的可视化展示。词法分析结果以表格形式呈现，LL(1)分析和SLR(1)分析过程中，分析栈和输入指针的变化以图形方式动态展示，SLR(1)分析中的活前缀DFA也以图形化方式呈现。

**错误处理模块**实现了常见语法错误的检测与报告，包括缺少分号、括号不匹配、标识符未声明、重复声明等错误。错误信息包含行号和错误类型描述，便于用户定位和修正问题。

### 2.4 PL/0语言文法说明

PL/0语言的文法定义是本系统实现的基础。原始PL/0语言文法存在左递归性，直接进行自顶向下分析会导致无限递归。为了实现LL(1)分析，需要对原始文法进行预处理，消除左递归并提取左公因子。

**原始PL/0语言文法（部分）**：

```
程序 → 子程序 .
子程序 → [常量说明][变量说明][过程说明]语句
常量说明 → const 标识符 = 数字 ; 常量说明 | ε
变量说明 → var 标识符 ; 变量说明 | ε
过程说明 → procedure 标识符 ; 子程序 ; 过程说明 | ε
语句 → 标识符 := 表达式 | if 条件 then 语句 | while 条件 do 语句
       | call 标识符 | begin 语句 {; 语句} end | read 标识符 | write 表达式 | ε
条件 → odd 表达式 | 表达式 比较运算符 表达式
表达式 → [+|-] 项 {+|- 项}
项 → 因子 {*|/ 因子}
因子 → 标识符 | 数字 | ( 表达式 )
```

**化简后的LL(1)文法**（已消除左递归）：

```
Program → SubProgram .
SubProgram → ConstDecl VarDecl ProcDecl Stmt

ConstDecl → const ConstDef ; ConstDeclPrime | ε
ConstDeclPrime → const ConstDef ; ConstDeclPrime | ε
ConstDef → Ident = Number

VarDecl → var Ident ; VarDeclPrime | ε
VarDeclPrime → var Ident ; VarDeclPrime | ε

ProcDecl → procedure Ident ; SubProgram ; ProcDeclPrime | ε
ProcDeclPrime → procedure Ident ; SubProgram ; ProcDeclPrime | ε

Stmt → Ident := Expr | if Cond then Stmt | while Cond do Stmt
      | call Ident | read Ident | write Expr | begin Stmt StmtPrime end | ε
StmtPrime → ; Stmt StmtPrime | ε

Cond → odd Expr | Expr RelOp Expr

Expr → + Term ExprPrime | - Term ExprPrime | Term ExprPrime
ExprPrime → + Term ExprPrime | - Term ExprPrime | ε

Term → Factor TermPrime
TermPrime → * Factor TermPrime | / Factor TermPrime | ε

Factor → Ident | Number | ( Expr )

RelOp → = | # | < | <= | > | >=
```

化简后的文法消除了所有左递归，并按照LL(1)文法的要求进行了分解和重构，使得每个产生式的选择可以仅通过查看一个前看符号来确定。

## 第3章 详细设计与实现

### 3.1 词法分析模块

#### 3.1.1 词法分析器的功能与原理

词法分析是编译过程的第一个阶段，其任务是将源程序的字符序列转换为Token序列。Token是编译过程中最小的语义单元，包括单词符号和其属性值。词法分析器读取源程序文本，按照预定的词法规则进行扫描和识别，忽略空白字符和注释，最终输出一个Token序列供语法分析器使用。

本系统的词法分析器采用状态转换图（DFA）作为理论基础。状态转换图是一种有向图，节点表示分析过程中的状态，边表示在某种输入字符下进行的转移。词法分析器从初始状态出发，沿着字符流逐个读取字符，根据当前状态和输入字符进行状态转移，当到达接受状态时输出一个识别到的Token。

#### 3.1.2 单词分类表设计

PL/0语言的单词符号分为以下几类：关键字是语言预先定义的具有特殊含义的单词，包括const、var、procedure、begin、end、if、then、while、do、call、read、write、odd等；标识符是用户定义的变量名、过程名等，以字母开头后跟字母或数字的字符串；常数是整型常数，仅包含数字字符；运算符包括赋值运算符（:=）、比较运算符（=、#、<、<=、>、>=）和算术运算符（+、-、*、/）；界限符包括分号（;）、逗号（,）、句点（.）、左括号（(）、右括号（））。

表3-1给出了单词分类的详细定义。

**表3-1 PL/0语言单词分类表**

| 类别 | 单词符号 | 属性值说明 |
|------|----------|------------|
| 关键字 | const, var, procedure, begin, end, if, then, while, do, call, read, write, odd | 关键字本身 |
| 标识符 | 字母开头的字母数字串 | 标识符字符串 |
| 常数 | 数字串 | 整数值 |
| 运算符 | :=, =, #, <, <=, >, >=, +, -, *, / | 运算符本身 |
| 界限符 | ;, ,, ., (, ) | 界限符本身 |

#### 3.1.3 状态转换图设计

词法分析器针对每类单词设计了相应的状态转换图。以标识符为例，状态转换图如下：状态0为初始状态，当读取到字母时转移到状态1；状态1为已识别前缀状态，当继续读取字母或数字时保持在状态1，当读取到其他字符时转移到状态2（接受状态）。数字的状态转换图类似，状态0读取数字后转移到状态1，状态1继续读取数字保持在状态1。

运算符和界限符大多为单个字符或双字符（如:=），通过简单的字符匹配即可识别。

#### 3.1.4 词法识别流程

词法分析器的工作流程如下：首先跳过空白字符（包括空格、制表符、换行符）；然后跳过注释行（以//开头的字符序列）；接着根据首字符判断可能的Token类型：如果首字符为字母，则进入标识符/关键字识别流程；如果首字符为数字，则进入数字识别流程；如果首字符为:、<、>，则尝试识别双字符运算符；否则按照单字符运算符或界限符处理。

在识别过程中，词法分析器使用DFA对标识符和数字进行验证，确保其符合PL/0语言的词法规则。识别到完整的Token后，将其加入Token序列，继续扫描后面的字符。

#### 3.1.5 核心代码实现

Token结构体的定义如下：

```cpp
struct Token {
    TokenType type;      // Token类型
    string value;        // Token值
    int line;            // 行号
    
    Token() : type(TokenType::ERROR), value(""), line(0) {}
    Token(TokenType t, const string& v, int l)
        : type(t), value(v), line(l) {}
};
```

词法分析器的核心函数getNextToken实现如下：

```cpp
Token Lexer::getNextToken() {
    skipWhitespace();
    
    if (isalpha(currentChar_)) {
        return analyzeIdentifier();
    }
    if (isdigit(currentChar_)) {
        return analyzeNumber();
    }
    // 处理运算符和界限符...
}
```

#### 3.1.6 词法分析结果输出格式

词法分析结果以表格形式输出，每行包含Token的类型、值和行号。表头为“类型 | 值 | 行号”，后续各行为实际的Token信息。输出示例如下：

```

IDENT       | n          | 1
OPERATOR    | =          | 1
NUMBER      | 10         | 1
DELIMITER   | ;          | 1
...
```

### 3.2 LL(1)语法分析模块

#### 3.2.1 LL(1)分析法原理

LL(1)分析法是一种自顶向下的语法分析方法，其中第一个L表示从左到右扫描输入，第二个L表示最左推导，括号中的1表示只向前看一个符号即可做出分析决策。LL(1)分析法使用一个分析栈和一个输入缓冲区，通过栈操作模拟最左推导过程。

LL(1)文法的判别条件是：对于文法中每个非终结符的各个产生式，其SELECT集两两不相交。SELECT集的计算公式为：如果产生式右部不以ε开头，则SELECT(A→α) = FIRST(α)；如果产生式右部以ε开头，则SELECT(A→α) = FOLLOW(A)。

#### 3.2.2 文法预处理

原始PL/0语言文法存在左递归性，例如Expr → Expr + Term会导致递归下降分析陷入无限递归。为了进行LL(1)分析，需要消除左递归。

消除左递归的方法是将左递归产生式转换为等价的右递归形式。以表达式为例：

```
原文法：
Expr → Expr + Term | Expr - Term | Term

消除左递归后：
Expr → Term ExprPrime
ExprPrime → + Term ExprPrime | - Term ExprPrime | ε
```

类似地，还需提取左公因子以确保LL(1)分析表的唯一性。

#### 3.2.3 集合计算

FIRST集的定义为：FIRST(α)表示可以由字符串α推导出的首个终结符的集合。计算FIRST集的算法如下：首先将所有终结符的FIRST集初始化为其自身；对于包含ε的产生式，将ε加入其左部非终结符的FIRST集；反复应用规则直到FIRST集不再变化。

FOLLOW集的定义为：FOLLOW(A)表示在某些句型中可能出现在非终结符A右边的终结符的集合。计算FOLLOW集的算法如下：将#加入开始符号的FOLLOW集；对于形如B → αAβ的产生式，将FIRST(β)中的非ε元素加入FOLLOW(A)；对于形如B → αA或B → αAβ（β⇒*ε）的产生式，将FOLLOW(B)加入FOLLOW(A)。

例如，对于化简后的文法，计算得到的FIRST集和FOLLOW集部分结果如下：

FIRST(Stmt) = {Ident, if, while, call, read, write, begin, ε}
FIRST(Expr) = {+, -, Ident, Number, (}
FOLLOW(Stmt) = {#, ;, end}
FOLLOW(Expr) = {#, ;, ), +, -, *, /, =, #, <, <=, >, >=}

#### 3.2.4 LL(1)分析表的自动生成

LL(1)分析表是一个二维表，行表示非终结符，列表示终结符，表项内容为产生式或错误标志。生成LL(1)分析表的算法如下：对于文法的每个产生式A → α，计算其SELECT集；对于SELECT集中的每个终结符a，将产生式填入分析表M[A, a]；对于SELECT集中的ε，用FOLLOW(A)中的终结符填充表项。

表3-2给出了部分LL(1)分析表的内容示例。

**表3-2 LL(1)分析表示例（部分）**

|----------|-------|-------|-------|-----|---|---|
| Stmt | Ident:=Expr | - | begin... | if... | ε | - |

#### 3.2.5 表驱动分析算法实现

表驱动LL(1)分析算法的伪代码如下：

```
将#和开始符号S入栈
从输入缓冲区读取第一个Token
repeat
    X = 栈顶符号
    a = 当前输入符号
    if X是终结符或# then
        if X == a then
            弹栈，读取下一个Token
        else error()
    else    // X是非终结符
        if M[X,a]是产生式 then
            弹栈
            将产生式右部逆序入栈
        else error()
until X == # and a == #
```

分析过程中，栈的变化和输入指针的位置被记录下来，用于后续的可视化展示。

#### 3.2.6 分析栈可视化设计

分析栈的可视化采用EasyX图形库实现。界面布局包括：左侧显示分析栈的内容，栈底在下，栈顶在上；中间显示当前使用的产生式；右侧显示剩余输入。栈内容用矩形框表示，每个符号占一个框，框内显示符号名称。当进行推导时，新产生的符号以动画方式入栈，被匹配的符号以动画方式出栈。

#### 3.2.7 错误检测与处理机制

LL(1)分析器检测的错误主要包括：词法错误（在识别标识符或数字时）、语法错误（分析表项为空或发生冲突）。错误处理采用同步恢复策略，当检测到错误时，跳过输入中的一些符号直到找到同步集合中的元素，然后从相应的非终结符重新开始分析。

### 3.3 SLR(1)语法分析模块

#### 3.3.1 SLR(1)分析法原理

SLR(1)分析法是一种自底向上的LR分析方法的简化形式。LR分析方法从左到右扫描输入，进行最右推导的逆过程。SLR(1)中的S表示Simple，表明它使用Follow集来帮助解决归约-归约冲突。

SLR(1)分析器通过构造项目集规范族和活前缀DFA来描述分析状态，使用分析表（ACTION表和GOTO表）来指导分析动作。分析过程中，维护一个状态栈和一个符号栈，根据当前状态和输入符号决定是移进、归约还是接受。

#### 3.3.2 拓广文法

为了便于构造分析表，需要为原文法添加一个产生式形成拓广文法。对于PL/0文法，添加产生式S' → Program，其中S'是新的开始符号，不出现在原文法的右部。

拓广文法的作用是：当归约到S'时，表示整个输入已被成功识别，可以进行接受操作。

#### 3.3.3 项目集规范族的生成

LR项目是产生式加上一个圆点位置，表示分析过程中已识别的部分。CLOSURE函数用于计算一个项目集的闭包，算法如下：如果项目A → α·Bβ在闭包中，且B → γ是文法产生式，则项目B → ·γ也在闭包中；重复直到没有新项目加入。

GO函数用于计算项目集I在符号X上的转移，算法如下：对于I中的每个项目A → α·Xβ，将项目A → αX·β加入结果集，然后对结果集取闭包。

项目集规范族是所有通过CLOSURE和GO函数能够到达的项目集的集合。

例如，对于产生式Expr → Term ExprPrime，其对应的项目表示为Expr → ·Term ExprPrime（初始项目）、Expr → Term·ExprPrime（部分匹配）、Expr → Term ExprPrime·（归约项目）。

#### 3.3.4 活前缀DFA的构造与可视化

活前缀DFA是项目集规范族与状态转移关系的图形化表示。每个状态是一个项目集，状态之间的边用文法符号标记。

活前缀DFA的可视化采用EasyX图形库实现。每个状态用一个矩形框表示，框内列出该状态中的所有项目。状态之间的转移用带箭头的线表示，线上标注转移符号。初始状态用特殊标记（如双边框）标识。

#### 3.3.5 SLR(1)分析表的自动生成

SLR(1)分析表由ACTION表和GOTO表组成。ACTION表用于记录终结符上的操作，包括移进（sn，表示将状态n入栈）、归约（rn×m，表示用第m条产生式归约）和接受（acc）；GOTO表用于记录非终结符上的状态转移。

构造SLR(1)分析表的算法如下：对于项目集I中的每个项目A → α·aβ（a为终结符），如果GO(I, a) = J，则在ACTION[I, a]填入移进操作；对于项目集I中的每个归约项目A → α·，如果A ≠ S'，则对于FOLLOW(A)中的每个终结符a，在ACTION[I, a]填入归约操作；对于接受项目S' → S·，在ACTION[I, #]填入接受操作；对于非终结符X，如果GO(I, X) = J，则在GOTO[I, X]填入状态J。

#### 3.3.6 移进归约分析算法实现

SLR(1)分析算法的伪代码如下：

```
将初始状态0入状态栈，将#入符号栈
repeat
    s = 状态栈栈顶
    a = 当前输入符号
    if ACTION[s, a] = sn then
        将a和n分别入符号栈和状态栈
        读取下一个输入符号
    else if ACTION[s, a] = rn then
        用第n条产生式A → β归约
        弹出|β|个符号和状态
        t = 状态栈栈顶
        将A和GOTO[t, A]分别入符号栈和状态栈
    else if ACTION[s, a] = acc then
        接受，分析成功
    else error()
until 接受或错误
```

#### 3.3.7 双栈可视化设计

SLR(1)分析过程涉及状态栈和符号栈两个栈，可视化界面同时展示两个栈的内容。界面左侧显示状态栈，状态以数字形式展示；中间显示符号栈，符号以字符串形式展示；右侧显示当前分析动作（移进/归约/接受）和使用的产生式。

栈的变化以动画形式呈现：移进时，新状态和符号从右侧滑入栈中；归约时，弹出的符号和状态以动画形式从栈中移除，归约后的非终结符和新状态从右侧滑入。

### 3.4 语法分析树可视化（加分项）

#### 3.4.1 树节点结构

语法分析树的节点结构定义如下：

```cpp
struct TreeNode {
    string label;                    // 节点标签
    vector<TreeNode*> children;      // 子节点指针向量
    int level;                       // 节点层次
    int position;                    // 节点位置
    
    TreeNode(const string& l) : label(l), level(0), position(0) {}
};
```

每个节点包含标签（存储文法符号）和子节点指针。树的层次和位置信息用于后续的图形布局计算。

#### 3.4.2 递归绘制算法

语法分析树的绘制采用递归算法。从根节点开始，先计算当前节点的图形位置，然后递归绘制所有子节点，最后用线段连接父节点和子节点。

布局算法采用分层布局策略：首先计算树的深度和各节点的层次；然后在同一层次内平均分配节点位置；最后根据节点位置计算绘制坐标。

#### 3.4.3 界面实现

语法分析树可视化界面采用EasyX图形库实现。界面中央显示语法分析树，节点用椭圆表示，椭圆内显示符号名称。父子节点之间用线段连接。界面底部提供操作按钮，包括重新绘制、放大、缩小、平移等功能。

set<LRItem> closure(const set<LRItem>& items) {
    set<LRItem> result = items;
    bool changed = true;
    while (changed) {
        changed = false;
        for (const auto& item : result) {
            if (item.dotPos < item.prod.right.size()) {
                string B = item.prod.right[item.dotPos];
                if (g.isNonTerm(B)) {
                    for (const auto& p : g.getProds()) {
                        if (p.left == B) {
                            LRItem newItem(p, 0);
                            if (!result.count(newItem)) {
                                result.insert(newItem);
                                changed = true;
                            }
                        }
                    }
                }
            }
        }
    }
    return result;
}
